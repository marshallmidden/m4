### Design Decision

The key insight: `sleep` commands in .fs files are what advance time. In real-time mode, `sleep` calls `fluid_msleep()` which sleeps the thread. For file rendering, we need to render audio frames instead of sleeping.

Approach:
1. Add a `fluid_file_renderer_t *renderer` field to the command handler struct
2. When `renderer` is set, `sleep` computes `ms * sample_rate / 1000` samples and calls `fluid_file_renderer_process_block()` instead of `fluid_msleep()`
3. In `fluidsynth.c`, when `-F` is used without a MIDI file, create a file renderer, attach it to the handler, and run the shell

#### Task 2: Modify fluid_handle_sleep to render audio frames

**`src/bindings/fluid_cmd.c`** -- Modified `fluid_handle_sleep`:
```c
int
fluid_handle_sleep(void *data, int ac, char **av, fluid_ostream_t out)
{
    FLUID_ENTRY_COMMAND(data);

    if(ac < 1)
    {
        fluid_ostream_printf(out, "sleep: too few arguments.\n");
        return -1;
    }

    if(!fluid_is_number(av[0]))
    {
        fluid_ostream_printf(out, "sleep: argument should be a number in ms.\n");
        return -1;
    }

    if(handler->renderer != NULL)
    {
        /* File rendering mode: render audio frames instead of sleeping */
        double ms = atof(av[0]);
        double sample_rate;
        int period_size;

        fluid_settings_getnum(handler->settings, "synth.sample-rate", &sample_rate);
        fluid_settings_getint(handler->settings, "audio.period-size", &period_size);

        handler->pending_samples += ms * sample_rate / 1000.0;

        while(handler->pending_samples >= period_size)
        {
            if(fluid_file_renderer_process_block(handler->renderer) != FLUID_OK)
            {
                fluid_ostream_printf(out, "sleep: file renderer error.\n");
                return -1;
            }

            handler->pending_samples -= period_size;
        }

        return 0;
    }

    fluid_msleep(atof(av[0]));    /* delay in milliseconds */

    return 0;
}
```
